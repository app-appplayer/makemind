A Dart and Flutter package family for the Model Context Protocol.

The packages compose into a single stack: bundle schema and standard ports, MCP client/server with LLM provider integration, a knowledge subsystem, transport-agnostic IO adapters, domain helpers, a Material 3 UI runtime, and an application orchestrator.

Overview

An MCP server publishes tools, resources, and a UI definition. A client connects through one of the supported transports and renders the UI; tool invocations originate from UI actions. The packages on this site provide the parts that make this loop work — protocol implementations, the bundle and port contract that lets servers describe themselves uniformly, IO adapters that bring physical and network devices under that contract, knowledge subsystems for fact recording and skill execution, and a Flutter UI runtime that renders MCP-served definitions.

Each package is published independently on pub.dev and can be used in isolation. The parts are designed to compose without ceremony.

Architecture

Lower layers do not depend on higher layers. Most packages depend only on mcp_bundle.

Composition

The packages do not require each other to be used together. Common compositions:

  • Minimal client — mcp_client alone is enough to talk to any MCP server.
  • Server with UI — mcp_server plus mcp_bundle describes tools and an MCP UI definition; clients render it through flutter_mcp_ui_runtime.
  • Knowledge stack — mcp_knowledge composes mcp_fact_graph + mcp_skill + mcp_profile + mcp_knowledge_ops + mcp_philosophy behind one facade.
  • Device control — mcp_io with one or more transport adapters (websocket, http, mqtt, serial, can, modbus, opcua, scpi) brings devices under a single contract; mcp_flow_runtime orchestrates control flows.
  • Multi-node deployment — mcp_gateway routes between MCP servers and clients with namespace separation and policy.
  • Application host — appplayer_core wires connection lifecycle, sessions, and bundle install on top of mcp_client, mcp_bundle, and the UI runtime.

Most packages depend only on mcp_bundle. There are no circular dependencies.
